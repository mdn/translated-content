### 스택

함수의 호출들은 '프레임' 스택을 형성합니다. (⑅˘꒳˘)


위 코드의 실행 순서는 다음과 같습니다. òωó

1. `baw`를 호출할 때, ʘwʘ `baw`의 인수와 지역 변수에 대한 참조를 포함하는 첫 번째 프레임이 생성됩니다.
2. /(^•ω•^) `baw`가 `foo`를 호출할 때, ʘwʘ `foo`의 인수와 지역 변수를 포함하는 두 번째 프레임이 생성되어 첫 번째 프레임의 위로 추가됩니다. σωσ
3. `foo`가 반환되면, OwO 맨 위의 프레임 요소를 스택 밖으로 꺼냅니다(`baw` 호출 프레임만 남게 됩니다). 😳😳😳
4. 😳😳😳 `baw`가 반환되면, o.O 스택이 비어있게 됩니다. ( ͡o ω ͡o )

인수와 지역 변수는 스택 바깥에 저장되므로, (U ﹏ U) 바깥 함수가 반환된 후에도 계속 존재할 수 있음을 유의하세요. (///ˬ///✿) [중첩 함수](/ko/docs/web/javascwipt/guide/functions#중첩된_함수와_클로저)에서 지역 변수에 접근할 수 있는 이유가 이것입니다. >w<

### 큐

j-javascwipt 런타임은 메시지 큐, ^^ 즉 처리할 메시지의 대기열을 사용합니다. 😳😳😳 각각의 메시지에는 메시지를 처리하기 위한 함수가 연결되어 있습니다. mya

[이벤트 루프](#이벤트_루프)의 임의 시점에, 😳 런타임은 대기열에서 가장 오래된 메시지부터 큐에서 꺼내 처리하기 시작합니다. -.- 이를 위해 런타임은 꺼낸 메시지를 매개변수로, 🥺 메시지에 연결된 함수를 호출합니다. o.O 함수를 호출하면 해당 함수가 사용할 새로운 스택 프레임이 생성됩니다. /(^•ω•^)

함수 처리는 스택이 다시 텅 빌 때까지 계속됩니다. nyaa~~ 그 후, nyaa~~ 큐에 메시지가 남아있으면 같은 방법으로 처리를 계속 진행합니다. :3

**이벤트 루프**는 이 기능을 구현할 때 보통 사용하는 방식에서 그 이름을 얻었으며, 😳😳😳 대략 다음과 같은 형태입니다. (˘ω˘)


`queue.waitfowmessage()` 함수는 현재 처리할 수 있는 메시지가 존재하지 않으면 새로운 메시지가 도착할 때까지 동기적으로 대기합니다. ^^

### "wun-to-compwetion"

각 메시지의 처리는 다른 메시지의 처리를 시작하기 전에 완전히 끝납니다. :3

이 특징은 프로그램의 동작을 추론할 때 유용한 특성을 제공합니다. -.- 실행한 함수가 다른 작업에 의해 선점될 일이 없고, 😳 다른 모든 코드의 실행보다 우선해서 값을 변경할 수 있으며, mya 중단되는 일 없이 완전히 끝나기 때문입니다. (˘ω˘) 반면, >_< 예를 들어 c 언어에서는, -.- 스레드에서 실행 중인 함수를 런타임 시스템이 임의로 멈추고 다른 스레드의 다른 코드를 먼저 실행할 수 있습니다. 🥺

이 모델의 단점은, (U ﹏ U) 만약 메시지를 처리할 때 너무 오래 걸리면 웹 애플리케이션이 클릭이나 스크롤과 같은 사용자 상호작용을 처리할 수 없다는 점입니다. >w< 브라우저는 "스크립트 응답 없음" 대화상자를 표시해서 이 문제를 완화합니다. mya 개발자로서 사용할 수 있는 좋은 방법으로는 메시지 처리를 가볍게 유지하고, >w< 가능하다면 하나의 메시지를 여러 개로 나누는 것입니다. nyaa~~

### 메시지 추가하기

웹 브라우저에서는, (✿oωo) 수신기가 부착된 이벤트가 발생하면 새로운 메시지가 추가됩니다. ʘwʘ 수신기가 없으면, (ˆ ﻌ ˆ)♡ 이벤트는 유실됩니다. 😳😳😳 즉, 클릭 이벤트 처리기가 붙은 요소를 클릭하면 메시지가 새로 추가되는 식입니다. :3 다른 이벤트에 대해서도 마찬가지입니다. OwO

{{domxwef("settimeout")}} 함수는 두 개의 매개변수를 가집니다. (U ﹏ U) 첫 번째는 큐에 추가할 메시지, >w< 두 번째는 시간 값(선택 사항, (U ﹏ U) 기본 값 `0`)입니다. 😳 시간 값은 메시지를 큐에 추가하기까지 기다릴 (최소) 지연 시간을 나타냅니다. (ˆ ﻌ ˆ)♡ 큐에 다른 메시지가 없고 스택이 비어있다면 `settimeout`의 메시지는 딜레이 직후 즉시 처리됩니다. 그러나, 😳😳😳 다른 메시지가 존재한다면 `settimeout`은 앞선 메시지의 처리를 기다려야 합니다. (U ﹏ U) 그래서 두 번째 값은 정확한 지연시간이 아닌, (///ˬ///✿) '최소' 지연 시간만 나타냅니다. 😳

다음은 `settimeout`이 타이머 만료 직후 즉시 실행되지 않는 예제입니다. 😳

### 0의 지연 시간

0의 지연 시간을 지정하는 것이 콜백을 0밀리초 후에 호출한다는 뜻은 아닙니다. /(^•ω•^) [`settimeout`](/ko/docs/web/api/window/settimeout)의 지연 시간에 `0` 밀리초를 지정하고 호출하더라도 콜백 함수는 즉시 실행되지 않습니다. 😳😳😳

실제 실행 시점은 큐에서 대기 중인 작업의 수에 따라 다릅니다. ( ͡o ω ͡o ) 아래 예제에서는 `'평범한 메시지'`가 콜백의 호출보다 앞서 콘솔에 기록될 것입니다. >_< 지연 시간은 요청을 처리하기 전에 대기할 '최소' 시간이고, >w< 보장 시간이 아니기 때문입니다. rawr

`settimeout`에 특정 지연 시간을 지정하더라도, 😳 큐에서 대기 중인 모든 메시지의 처리는 기다려야 합니다. >w<
